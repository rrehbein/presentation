SOLID

SOLID

A collection of complementary principles for helping to design Object Oriented Architectures or Libraries
  • Single Responsibility Principle
  • Open / Closed Principle
  • Liskov's Substitution Principle
  • Interface Segregation Principle
  • Dependency Inversion Principle

S - Single Responsibility Principle

Classes and objects should have only one job

S - Single Responsibility Principle

A class should only have one "job" or "responsibility"
  • A reason for changing
A cache should worry about caching, not data encoding
A form should worry about validity, but not how to validate

A class should only have one "responsibility" or "job"
  • A reason for changing
Don't go OCD on it. 
$response = $service->process($request);

I - Interface Segregation Principle

Small, compact interfaces are preferable to large, 'God' objects

I - Interface Segregation Principle

Interface

  • An interface is a collection of methods
  • An interface is a promise to fulfil

I - Interface Segregation Principle

  • Honor the Single Responsibility Principle
  • Narrow collection of methods to allow performing a responsibility
interface MyCache {
    public function get($key);
    public function set($key, $value);
    public function has($key);
    public function delete($key);
}

L - Liskov's Substitution Principle

Objects of the same type should be interchangeable

L - Liskov's Substitution Principle

Type

  • An interface can be used to define a 'type' of object
  • A class can be used to define a 'type' of object
  • A parent class can be used to define a 'type' of object

L - Liskov's Substitution Principle

Interchange

  • An object's methods can be called in the same way as another of its type
  • An object accepts the same types of parameters
  • An object returns the same types of objects

L - Liskov's Substitution Principle

Objects of the same type should be interchangeable
$service->setCache($memcache);
$service->setCache($filecache);

O - Open / Closed Principle

Objects should be open to extension, but closed to modification

O - Open / Closed Principle

Extension without Modification

  • Polymorphic
    • Same interface
    • Different implementation
  • Like Liskov's Substitution Principle

O - Open / Closed Principle

class MyFileCache implements MyCache {
    public function __construct($path) { ... }
    public function gc() { ... }

    // and the MyCache methods
    public function get($key) { ... }
    public function set($key, $value) { ... }
    public function has($key) { ... }
    public function delete($key) { ... }
}

D - Dependency Inversion Principle

Applications should depend upon abstractions, not concrete instances of an object

D - Dependency Inversion Principle

Abstractions

  • An abstract class
  • An interface
  • A 'type' of object

D - Dependency Inversion Principle

Inversion

  • A reversal of the normal order
  • In the context of SOLID, objects should depend on abstractions

Injection

  • The act of injecting or throwing in forcible
  • Not actually part of SOLID

D - Dependency Inversion Principle

class MyLoggingCache implements MyCache {
    public function __construct(
        MyCache $realCache
    ) {
        $this->cache = $realCache;
        // UGLY: Depends on a concrete logger
        $this->logger = new MySyslogLogger();
    }
    ...
}

D - Dependency Inversion Principle

class MyLoggingCache implements MyCache {
    public function __construct(
        MyCache $realCache, MyLogger $logger
    ) {
        $this->cache = $realCache;
        // The logger is injected in
        $this->logger = $logger;
    }
    ...
}

D - Dependency Inversion Principle

with Single Responsibility Principle

  • Object creation depends on concrete classes
  • Object creation is a job or responsibility. Therefore things that create objects shouldn't do much else.
  • Object creation should be for factories or for controllers in MVC

D - Dependency Inversion Principle

Service Locators (SL)

  • Using an SL pattern means we depend on a locator

Dependency Injection Containers (DIC)

  • Using a DIC pattern means the container must tell an object everything it may need

D - Dependency Inversion Principle

SL vs DIC

class MyClass {
    public function __construct(
        ServiceContainer $cont
    ) {
        $this->cache  = $cont->get('cache');
        $this->logger = $cont->get('logger');
    }
}

D - Dependency Inversion Principle

SL vs DIC

class MyClass {
    public function __construct(
        MyCache $realCache, MyLogger $logger
    ) {
        $this->cache  = $realCache;
        $this->logger = $logger;
